# How We Built the Chainlink Integration in BlackIn

So let me just walk you through how this whole thing actually works, because looking at it from the outside it can seem like a lot of moving pieces. But once you trace through it from the moment a user clicks something on the screen to when a workflow is actually live on chain, it all starts to make sense.

It starts from the backend. The very first thing we had to figure out was how to treat Base Sepolia and Base Mainnet as real targets that Chainlink's CRE tooling would understand. So inside the adapter, there is a block that reads environment variables for the CLI path, the API key, the private key, the runner mode, and the chain selector. All of that normalization happens at [cre_adapter.ts L157-L221](https://github.com/Black-in/lighthouse-main/blob/aa832709ca3e80b0a2ac063167a2e9e43bf0bbc8/apps/server/src/chains/base/cre_adapter.ts#L157-L221). The reason we did it this way is because every single operation that comes after it, whether that is simulation or full deployment, needs to speak the same language. Without that normalized setup, you would end up patching environment values in five different places and things would start breaking in unpredictable ways.

Once the environment is sorted, the adapter moves on to actually building the Chainlink CRE workflow source. This is probably the most important part of the whole thing. The code writes a `package.json` that explicitly lists `@chainlink/cre-sdk` and `@chainlink/cre-sdk-javy-plugin` as dependencies. Then it writes a TypeScript `main.ts` that imports `Runner`, `CronCapability`, `EVMClient`, `HTTPClient`, and `ConfidentialHTTPClient` from the CRE runtime. The cron trigger is what kicks execution off. Then `EVMClient` reads state from the deployed Base contract, `ConfidentialHTTPClient` pulls in protected inputs for compliance checks, and `HTTPClient` fetches whatever market data is needed. All of this generation logic lives at [cre_adapter.ts L411-L540](https://github.com/Black-in/lighthouse-main/blob/aa832709ca3e80b0a2ac063167a2e9e43bf0bbc8/apps/server/src/chains/base/cre_adapter.ts#L411-L540). This is the actual core of why Chainlink matters in BlackIn, because it is not just importing the SDK, it is generating real workflows that use CRE capabilities.

After the source is generated, there is a dependency preparation step. Before we can run anything, the adapter needs to make sure the WASM plugin artifact under `@chainlink/cre-sdk-javy-plugin` is actually present. If it is not, it runs `bun x cre-setup` to get everything in place. It also handles both prebuilt and dynamic dependency modes so the setup works cleanly across different environments. That all happens at [cre_adapter.ts L995-L1103](https://github.com/Black-in/lighthouse-main/blob/aa832709ca3e80b0a2ac063167a2e9e43bf0bbc8/apps/server/src/chains/base/cre_adapter.ts#L995-L1103). Think of this block as the bridge between the generated source code and the moment you can actually execute something.

Now there are two paths from here. The first is simulation. When you are running in simulate-only mode, the adapter spins up a temporary CRE project with the generated files and runs `cre workflow simulate` with `--target`, `--non-interactive`, and `--trigger-index 0`. It gives you back a workflow summary, preflight status, and trimmed execution logs, and anything that would require a real deployment just gets marked as skipped. This is the path we used for hackathon demos before full deploy access was set up. That simulation path is at [cre_adapter.ts L1325-L1405](https://github.com/Black-in/lighthouse-main/blob/aa832709ca3e80b0a2ac063167a2e9e43bf0bbc8/apps/server/src/chains/base/cre_adapter.ts#L1325-L1405).

The second path is the full production deployment. Here the adapter runs strict preflight checks first, then deploys the Solidity contract using Foundry, then runs `cre workflow simulate`, `cre workflow deploy`, and `cre workflow activate` in sequence. It captures results from each step and assembles a final metadata object containing the workflow id, network target, chain selector, contract address, registry transaction, explorer URLs, and logs. That entire production lifecycle is at [cre_adapter.ts L1407-L1677](https://github.com/Black-in/lighthouse-main/blob/aa832709ca3e80b0a2ac063167a2e9e43bf0bbc8/apps/server/src/chains/base/cre_adapter.ts#L1407-L1677). This is the real thing, this is what actually puts a workflow on chain.

All of this adapter work sits behind a queue. The runtime worker is what consumes `cre-deploy` jobs and actually calls into the adapter. It enforces that only Base chains get routed here, checks whether simulate-only mode is active and whether an API key is available, publishes terminal log lines back to the user session in real time, and then writes the final deployment state into the database. In simulate mode it calls `executeCreGenerationWorkflow` and stores simulation metadata. In deploy mode it calls `executeCreDeployWorkflow` and stores the contract address and transaction data. The full wiring for that is across [cre_runtime_worker.ts L18-L57](https://github.com/Black-in/lighthouse-main/blob/aa832709ca3e80b0a2ac063167a2e9e43bf0bbc8/apps/server/src/queue/cre_runtime_worker.ts#L18-L57), [cre_runtime_worker.ts L108-L122](https://github.com/Black-in/lighthouse-main/blob/aa832709ca3e80b0a2ac063167a2e9e43bf0bbc8/apps/server/src/queue/cre_runtime_worker.ts#L108-L122), [cre_runtime_worker.ts L129-L269](https://github.com/Black-in/lighthouse-main/blob/aa832709ca3e80b0a2ac063167a2e9e43bf0bbc8/apps/server/src/queue/cre_runtime_worker.ts#L129-L269), and [cre_runtime_worker.ts L271-L291](https://github.com/Black-in/lighthouse-main/blob/aa832709ca3e80b0a2ac063167a2e9e43bf0bbc8/apps/server/src/queue/cre_runtime_worker.ts#L271-L291). This layer is what ties together the async execution, the live log streaming, and the persistence in one place.

When the server starts up, it runs CRE preflight automatically and then wires the `cre-deploy` queue and the runtime worker so everything is ready before any user request comes in. That startup wiring is at [init.ts L25-L37](https://github.com/Black-in/lighthouse-main/blob/aa832709ca3e80b0a2ac063167a2e9e43bf0bbc8/apps/server/src/services/init.ts#L25-L37) and [init.ts L42-L43](https://github.com/Black-in/lighthouse-main/blob/aa832709ca3e80b0a2ac063167a2e9e43bf0bbc8/apps/server/src/services/init.ts#L42-L43). And separately, the generator decides at code generation time whether to actually enqueue a CRE job, which it does by checking whether an API key is present or whether simulate-only mode is on. That decision lives at [generator.ts L180-L186](https://github.com/Black-in/lighthouse-main/blob/aa832709ca3e80b0a2ac063167a2e9e43bf0bbc8/apps/server/src/generator/generator.ts#L180-L186). The point is that from the moment a user hits generate, the system already knows whether to run simulation or deployment and queues accordingly.

There is also a template generator that makes sure every AI-generated workspace comes with a proper CRE folder already in it. It writes `cre/base-runtime-workflow` files, including workflow config for both staging and production, and embeds the CRE SDK dependency set into the generated project. So if you generate a new project today on BlackIn and look at the file tree, there is already a CRE workflow directory sitting there waiting. That template emission happens at [template.ts L234-L346](https://github.com/Black-in/lighthouse-main/blob/aa832709ca3e80b0a2ac063167a2e9e43bf0bbc8/apps/server/src/chains/base/template.ts#L234-L346).

We also built a health controller so anyone running the backend can actually see whether CRE is ready or not. It runs startup and strict preflight checks, verifies that both the CRE CLI and Forge are available, and returns a structured response that tells you whether you are in simulation-mode readiness or full deployment-mode readiness. If something is misconfigured in the environment, this endpoint will tell you exactly what. That is at [getCreHealthController.ts L60-L105](https://github.com/Black-in/lighthouse-main/blob/aa832709ca3e80b0a2ac063167a2e9e43bf0bbc8/apps/server/src/controllers/health-controller/getCreHealthController.ts#L60-L105).

The socket service is the gateway that connects what happens in the terminal on the frontend to the actual queue on the backend. When a user runs a Base deploy command in the terminal, the socket service picks that up, maps `lighthouse_DEPLOY_BASE_SEPOLIA` or `lighthouse_DEPLOY_BASE_MAINNET` to the right network values, validates credentials according to the current mode, and drops a `cre-deploy` job into the queue. That routing logic is spread across [services.command.ts L21-L37](https://github.com/Black-in/lighthouse-main/blob/aa832709ca3e80b0a2ac063167a2e9e43bf0bbc8/apps/socket/src/services/services.command.ts#L21-L37), [services.command.ts L91-L113](https://github.com/Black-in/lighthouse-main/blob/aa832709ca3e80b0a2ac063167a2e9e43bf0bbc8/apps/socket/src/services/services.command.ts#L91-L113), and [services.command.ts L134-L140](https://github.com/Black-in/lighthouse-main/blob/aa832709ca3e80b0a2ac063167a2e9e43bf0bbc8/apps/socket/src/services/services.command.ts#L134-L140). This is the piece that makes the terminal feel live, because the user types a command and the backend immediately knows to start a CRE job.

On the frontend side, the integration is much simpler but it still matters. The frontend does not touch the Chainlink SDK at all, but it is the thing that drives every user action that eventually ends up in the backend CRE path. The terminal command definitions that expose the Base deploy actions live at [terminal_commands.ts L35-L37](https://github.com/Black-in/lighthouse-club/blob/69cb2c68d8ed9379373f1bfbfcce3e67d0d8312e/apps/web/src/lib/terminal_commands.ts#L35-L37), [terminal_commands.ts L58-L59](https://github.com/Black-in/lighthouse-club/blob/69cb2c68d8ed9379373f1bfbfcce3e67d0d8312e/apps/web/src/lib/terminal_commands.ts#L58-L59), and [terminal_commands.ts L74-L77](https://github.com/Black-in/lighthouse-club/blob/69cb2c68d8ed9379373f1bfbfcce3e67d0d8312e/apps/web/src/lib/terminal_commands.ts#L74-L77). The terminal handler that actually sends those commands over the websocket is at [useTerminal.ts L69-L77](https://github.com/Black-in/lighthouse-club/blob/69cb2c68d8ed9379373f1bfbfcce3e67d0d8312e/apps/web/src/hooks/useTerminal.ts#L69-L77).

When the user hits generate from the UI, the generate API always sends `chain: BASE` in the request body, which is what aligns everything with the backend CRE adapter from the very beginning. That is at [generate_contract.ts L115-L127](https://github.com/Black-in/lighthouse-club/blob/69cb2c68d8ed9379373f1bfbfcce3e67d0d8312e/apps/web/src/lib/server/generate_contract.ts#L115-L127) and [generate_contract.ts L316-L319](https://github.com/Black-in/lighthouse-club/blob/69cb2c68d8ed9379373f1bfbfcce3e67d0d8312e/apps/web/src/lib/server/generate_contract.ts#L316-L319). And then the right panel UI lets users pick between Base Sepolia and Base Mainnet and shows them the deployment success metadata that came back from backend CRE execution. That panel component is at [RightPanelActions.tsx L192-L232](https://github.com/Black-in/lighthouse-club/blob/69cb2c68d8ed9379373f1bfbfcce3e67d0d8312e/apps/web/src/components/builder/RightPanelActions.tsx#L192-L232) with the network controls at [RightPanelActions.tsx L295-L310](https://github.com/Black-in/lighthouse-club/blob/69cb2c68d8ed9379373f1bfbfcce3e67d0d8312e/apps/web/src/components/builder/RightPanelActions.tsx#L295-L310).

So the whole thing is really a two layer model. The frontend is entirely about capturing intent, whether that is a user generating a contract, selecting a network, or typing a deploy command in the terminal. The backend is entirely about execution, meaning it generates the actual CRE workflow source, resolves dependencies, runs simulation or full deployment through the Chainlink CLI, captures all the metadata, and persists it. Every link above is part of that path and together they form the complete Chainlink integration as it stands in BlackIn today.
