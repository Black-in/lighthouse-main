## Backend and Frontend Integration Explanation

The backend begins by treating Base Sepolia and Base Mainnet as explicit CRE targets, then resolves CLI path, API key, private key, runner mode, and chain selector configuration from environment variables. This setup happens in the adapter so every subsequent CRE operation uses one normalized runtime contract. The same block also standardizes parsing of deploy metadata from CLI output so workflow id, transaction hash, explorer URL, binary URL, and config URL are captured in a structured way and passed downstream.

The next backend block generates the Chainlink CRE workflow source itself. The code writes a `package.json` that explicitly depends on `@chainlink/cre-sdk` and `@chainlink/cre-sdk-javy-plugin`, then writes a TypeScript `main.ts` workflow that imports CRE runtime primitives such as `Runner`, `CronCapability`, `EVMClient`, `HTTPClient`, and `ConfidentialHTTPClient`. In the generated workflow, the cron trigger drives execution, `EVMClient` reads onchain state from the deployed Base contract, `ConfidentialHTTPClient` loads protected inputs for compliance checks, and `HTTPClient` fetches market data. This is the core line-level implementation of Chainlink CRE capability usage in BlackIn.

After source generation, the adapter ensures workflow dependencies and the required WASM plugin are ready before execution. It verifies the CRE plugin artifact path under `@chainlink/cre-sdk-javy-plugin`, runs `bun x cre-setup` when artifacts are missing, and supports prebuilt or dynamic dependency modes so simulation and deployment can run reliably in different environments. This dependency preparation block is the bridge between generated workflow code and runnable CRE CLI execution.

For simulation mode, the adapter creates a temporary CRE project with generated workflow files and executes `cre workflow simulate` using `--target`, `--non-interactive`, and `--trigger-index 0`. This line path returns workflow summary, preflight status, and trimmed execution logs, and marks non simulation steps as skipped when running simulation-only mode. This is exactly the path used for local hackathon demonstration when deploy access is not yet enabled.

For full deployment mode, the adapter first runs strict CRE preflight checks, then deploys the Solidity contract with Foundry, then executes `cre workflow simulate`, `cre workflow deploy`, and `cre workflow activate` in order. It captures outcomes for each step, then assembles metadata containing workflow id, network target, chain selector, contract address, registry transaction, explorer URLs, and logs. This section is the production CRE lifecycle implementation and is the authoritative deploy path in BlackIn.

The runtime worker connects queue execution to this adapter. It consumes `cre-deploy` jobs, enforces Base-only routing, checks simulate-only mode and API key availability, publishes terminal logs back to the user session, and writes deployment state into the database. In simulate-only mode it runs `executeCreGenerationWorkflow` and stores simulation metadata. In deploy mode it runs `executeCreDeployWorkflow` and stores deployed contract address and transaction metadata. This is where asynchronous job execution, persistence, and user-visible progress are tied together.

Service bootstrap integrates CRE checks into server startup. During initialization, the backend runs CRE startup preflight, then wires `cre-deploy` queue and runtime worker so every generated Base contract can move into simulation or deployment flow without additional manual steps. The generator decides whether CRE jobs should be queued by evaluating API key presence or simulate-only mode, so prompt-to-workflow execution remains available in both deploy-enabled and simulation-only environments.

The template generator ensures new AI-generated workspaces include a complete CRE folder by default. It writes `cre/base-runtime-workflow` files, including workflow config for staging and production, and embeds the CRE SDK dependency set into generated project assets. This is why a generated BlackIn project includes a CRE workflow directory immediately after code generation.

The health controller exposes CRE readiness as an operational contract. It runs startup and strict preflight checks, verifies CRE CLI and Forge availability, and reports simulation-mode versus deployment-mode readiness in a structured health response. This gives operators a direct API view of whether Chainlink workflow execution is actually runnable in the current environment.

The socket command service is the command gateway between frontend and backend CRE operations. It maps `lighthouse_DEPLOY_BASE_SEPOLIA` and `lighthouse_DEPLOY_BASE_MAINNET` to Base network values, validates required CRE credentials according to simulate-only or deploy mode, and queues `cre-deploy` jobs. This is how user terminal actions are converted into the backend Chainlink workflow lifecycle.

The frontend does not import the Chainlink SDK directly, but it contains the user entry points that drive the backend Chainlink path. Terminal command definitions expose Base deploy actions, terminal handlers send deploy commands over websocket, and generate APIs always send `chain: BASE` so generated projects align with the backend CRE adapter expectations. The deploy panel UI also allows network selection for Base Sepolia and Base Mainnet and shows deployment success metadata that is produced by backend CRE execution.

Taken together, BlackIn uses a two layer model for Chainlink integration. The frontend is the interaction layer that requests Base generation and deployment actions, and the backend is the execution layer that composes CRE workflows, simulates them, deploys them, activates them, and persists auditable metadata. The linked files and lines above are the complete implementation path currently used in this project.
